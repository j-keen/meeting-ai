| 4 | 다단계 대화형 회의 준비 워크플로우 + 한국어 초성 검색 | ★★★☆ | ★★★☆ |

## 3. 특허 가능 발명 식별서

### 발명 1: Smart Token Strategy — 이전 요약 + 최근 발화 결합 분석 방법

#### 3.1.1 기술적 특징

**소스 위치**: `ai.js:19-45` (`buildTranscriptText` 함수)

실시간 회의 분석 시 LLM API의 토큰 제한과 비용 문제를 해결하기 위한 3단계 토큰 전략:

1. **Full Strategy**: 전체 회의록을 LLM에 전송 (정확도 최고, 비용 최대)
2. **Recent Strategy**: 최근 N분의 발화만 전송 (비용 절감, 맥락 손실 위험)
3. **Smart Strategy** (핵심 발명):
   - 이전 분석의 **요약(summary)**을 `[Previous Summary]` 블록으로 전달
   - 최근 N분의 **원문 발화**를 `[Recent Transcript]` 블록으로 결합
   - 두 블록을 하나의 프롬프트로 합성하여 LLM에 전송
   - 최근 발화가 없는 경우 마지막 20개 발화로 폴백

```javascript
// ai.js:38-45 - Smart Strategy 핵심 로직
if (previousSummary) {
  text += `[Previous Summary]\n${previousSummary}\n\n[Recent Transcript]\n`;
}
text += (recent.length > 0 ? recent : transcript.slice(-20))
  .map(formatLine).join('\n');
```

**연속 분석 체인**: `app.js:252-265`에서 이전 분석의 summary를 다음 분석에 자동 전달하여 누적적 맥락 유지:

```javascript
const previousSummary = state.analysisHistory.length > 0
  ? state.analysisHistory[state.analysisHistory.length - 1].summary
  : null;
```

#### 3.1.2 신규성 분석 (★★★★)

- 기존 회의 분석 도구(Otter.ai, Fireflies, Clova Note)는 회의 종료 후 전체 녹취록 기반 요약을 제공하거나, 실시간인 경우 전체 텍스트를 전송
- **이전 분석의 구조화된 요약 + 최근 원문 발화를 결합**하여 연속적 누적 분석을 수행하는 방식은 선행기술에서 확인되지 않음
- 특히 "분석 히스토리 체인"을 통한 맥락 연속성 유지 메커니즘이 차별점

#### 3.1.3 진보성 분석 (★★★★)

- 단순한 "요약 + 최근 텍스트 결합"이 아닌, **분석 결과의 구조화된 JSON 출력**(summary, context, openQuestions, actionItems, suggestions)을 다음 분석의 입력으로 재활용하는 **재귀적 분석 패턴**
- 설정 가능한 분석 주기(10~300초)와 결합하여, 토큰 효율성과 분석 정확도의 최적 균형을 자동으로 유지
- 통상의 기술자가 용이하게 도출하기 어려운 기술적 구성

### 발명 2: Function Calling 기반 회의 중 AI 양방향 상호작용 시스템

#### 3.2.1 기술적 특징

**소스 위치**: `chat.js:14-45, 271-289`

Gemini API의 Function Calling을 활용하여 AI가 회의 중 **능동적으로 시스템 상태를 변경**할 수 있는 3가지 함수 선언:

```javascript
const FUNCTION_DECLARATIONS = [
  { name: 'add_context',    // 사용자 인사이트를 분석 맥락에 추가
    description: 'Add user insight or context to enhance future analysis.' },
  { name: 'add_memo',       // 메모를 회의록에 자동 추가
    description: 'Add a memo note to the meeting transcript.' },
  { name: 'rerun_analysis', // AI 분석 재실행 트리거
    description: 'Re-run the AI analysis with accumulated context.' }
];
```

**핵심 메커니즘**:
1. 사용자가 AI 채팅으로 "이 부분 중요하니까 메모해줘"라고 요청
2. Gemini가 Function Calling으로 `add_memo` 함수 호출 결정
3. `handleFunctionCall()`이 실제로 앱 상태를 변경 (메모 추가, Pub/Sub 이벤트 발행)
4. UI가 실시간으로 업데이트됨

**양방향 맥락 흐름**:
- 채팅 시스템 프롬프트에 최근 회의록 20줄, 현재 분석 요약, 사용자 인사이트, 회의 배경을 자동 주입 (`chat.js:210-243`)
- AI가 회의 맥락을 이해한 상태에서 함수를 호출하므로, 맥락에 맞는 행동을 자율적으로 수행

#### 3.2.2 신규성 분석 (★★★☆)

- LLM Function Calling 자체는 공지기술
- 그러나 **회의 진행 중 실시간으로** AI가 회의 상태(메모, 분석 맥락, 재분석 트리거)를 변경하는 **양방향 상호작용 패턴**은 차별적
- 기존 도구(Otter, Fireflies)는 회의 후 정적 요약 제공에 그침

#### 3.2.3 진보성 분석 (★★★★)

- 회의 맥락(transcript + analysis + userInsights)을 채팅 시스템 프롬프트에 동적으로 주입하고, 그 맥락 하에서 Function Calling을 수행하여 시스템 상태를 변경하는 **폐쇄 루프(closed-loop) 인터랙션** 구조
- 특히 `rerun_analysis` 함수로 AI가 스스로 재분석을 트리거하여 누적된 맥락을 반영할 수 있는 점은 자동화된 회의 보조의 새로운 패러다임

### 발명 3: AI-사용자 하이브리드 오타 교정 사전 학습 시스템

#### 3.3.1 기술적 특징

**소스 위치**: `ai.js:194-228`, `app.js:213-234`, `app.js:738-754`, `ui.js:83-113`

3가지 학습 채널을 결합한 오타 교정 시스템:

**채널 1 — 사용자 수동 교정 학습** (`app.js:738-754`):
```javascript
on('transcript:edit', ({ id, text, original }) => {
  if (original && text !== original) {
    const origWords = original.split(/\s+/);
    const newWords = text.split(/\s+/);
    if (origWords.length === newWords.length) {
      for (let i = 0; i < origWords.length; i++) {
        if (origWords[i] !== newWords[i] && origWords[i].length > 1) {
          addTypoCorrection(origWords[i], newWords[i]); // 사전에 자동 등록
        }
      }
    }
  }
});
```

사용자가 회의록 텍스트를 인라인 편집하면, 단어 수준 변경을 감지하여 **오타 사전(typoDict)**에 자동 등록합니다.

**채널 2 — AI 주기적 교정 제안** (`ai.js:194-228`, `app.js:213-234`):
- 2분 주기로 최근 10개 발화를 Gemini API에 전송
- 기존 오타 사전을 함께 제공하여 중복 방지
- AI가 STT 오류 패턴을 인식하고 새로운 교정 규칙 제안
- 제안된 교정이 기존 사전과 병합되어 누적 학습

**채널 3 — 실시간 교정 적용** (`ui.js:89-100`):
```javascript
export function applyTypoCorrections(text) {
  let corrected = text;
  const corrections = [];
  for (const [before, after] of Object.entries(typoDict)) {
    if (corrected.includes(before)) {
      corrected = corrected.split(before).join(after);
      corrections.push({ before, after });
    }
  }
  return { text: corrected, corrections };
}
```

새로운 STT 결과가 들어올 때마다 축적된 오타 사전을 실시간 적용하고, 교정된 부분을 UI에서 점선 밑줄로 시각적 하이라이트 표시 (`styles.css:385-404`).

#### 3.3.2 신규성 분석 (★★★★)

- **선행기술 검토**:
  - US20070208567A1 (Error Correction in ASR Transcripts): 사용자 교정을 피드백하여 ASR 정확도 개선 → 서버 사이드 ASR 모델 재훈련 방식
  - US8019602B2 (ASR Learning Using User Corrections): 사용자 변경이 "마음 변경"인지 "인식 오류"인지 판별 → 음성 모델 자체 개선
  - 본 발명은 ASR 모델을 변경하지 않고, **텍스트 레벨에서 사전 기반 교정**을 수행하며, **AI와 사용자가 동시에 사전을 구축**하는 하이브리드 방식이라는 점에서 차별화

- **핵심 차별점**:
  1. 사용자 편집 + AI 주기적 제안의 **이중 학습 루프**
  2. 클라이언트 사이드 localStorage 기반 **개인화된 오타 사전 축적**
  3. STT 엔진 자체를 수정하지 않는 **비침습적(non-invasive) 교정**

#### 3.3.3 진보성 분석 (★★★★★)

- 단순한 사전 교체가 아닌, **사용자 행동(편집)으로부터의 암묵적 학습**과 **AI의 명시적 오류 패턴 분석**을 결합
- 회의가 진행될수록 STT 특화 오타 사전이 성장하여, 해당 회의/팀/도메인에 최적화된 교정 품질 달성
- 기존 선행기술에서 이러한 이중 학습 메커니즘의 결합은 발견되지 않음
- **가장 강한 진보성**을 가진 발명으로 평가

### 발명 4: 다단계 대화형 회의 준비 워크플로우 + 한국어 초성 검색

#### 3.4.1 기술적 특징

**소스 위치**: `meeting-prep.js` 전체, 특히 초성 검색 `meeting-prep.js:10-39`

**6단계 대화형 워크플로우**:
```
type → agenda → time → attendees → prompt → standby
```

각 단계에서 AI 챗봇 UI를 통해 사용자와 대화형으로 회의 설정을 수집하고, 최종 확인(standby) 후 자동 녹음 시작.

**한국어 초성(Chosung) 검색 알고리즘** (`meeting-prep.js:10-39`):
```javascript
const CHOSUNG = ['ㄱ','ㄲ','ㄴ','ㄷ','ㄸ','ㄹ','ㅁ','ㅂ','ㅃ','ㅅ','ㅆ','ㅇ','ㅈ','ㅉ','ㅊ','ㅋ','ㅌ','ㅍ','ㅎ'];

function getChosung(char) {
  const code = char.charCodeAt(0);
  if (code < 0xAC00 || code > 0xD7A3) return null;
  return CHOSUNG[Math.floor((code - 0xAC00) / 588)];
}
```

- 유니코드 한글 음절 분해를 통한 초성 추출
- `ㅎㄱㄷ` → "홍길동" 매칭 가능
- 일반 부분 문자열 검색과 초성 검색의 자동 판별 (`isAllChosung`)
- 참석자 연락처 검색에 적용

**추가 특징**:
- 프리셋 저장/로드: 회의 유형별 설정을 프리셋으로 저장하고 재사용
- 참석자 관리: 연락처 추가/선택/뱃지 표시 인터페이스
- 회의 시작 시 수집된 config(유형, 안건, 시간, 참석자, AI 프롬프트)를 분석 시스템에 자동 주입

#### 3.4.2 신규성 분석 (★★★☆)

- 대화형 설정 수집 자체는 챗봇 패턴으로 공지기술
- 그러나 **회의 준비 전용 6단계 워크플로우**에서 수집된 정보를 AI 분석 프롬프트에 **자동 주입**하는 엔드투엔드 연결은 차별적
- 한국어 초성 검색의 유니코드 분해 알고리즘 자체는 공지이나, 회의 참석자 검색에 적용한 것은 UX 차별화

#### 3.4.3 진보성 분석 (★★★☆)

- 워크플로우 + 초성 검색 + AI 프롬프트 자동 구성의 **결합**에서 진보성 인정 가능
- 단독으로는 진보성이 약하므로, 발명 1 또는 2의 종속항으로 기재하는 것이 전략적

### 발명 5: 클라이언트 전용 프라이버시 아키텍처 + 자동 정리 스토리지 시스템

#### 3.5.1 기술적 특징

**소스 위치**: `storage.js` 전체, `server.js` (정적 파일 서빙만 수행)

**완전한 클라이언트 사이드 처리**:
- 서버(`server.js`)는 34줄의 순수 정적 파일 서버 — 사용자 데이터를 전혀 수신/저장하지 않음
- 모든 회의 데이터(녹취록, 분석 결과, 채팅 기록)가 브라우저 localStorage에만 저장
- API 키도 Base64 인코딩하여 클라이언트에 저장 (`storage.js:115-127`)
- 외부 전송은 Gemini API 호출(분석/채팅)과 선택적 Slack 웹훅만

**localStorage 용량 관리 시스템** (`storage.js:7-55`):
```javascript
const MAX_MEETINGS = 50;
const WARN_THRESHOLD = 0.8;

function autoCleanup() {
  const data = loadAll();
  if (data.meetings.length > MAX_MEETINGS) {
    data.meetings.sort((a, b) => b.updatedAt - a.updatedAt);
    data.meetings = data.meetings.slice(0, MAX_MEETINGS);
  }
}
```

- 5MB localStorage 한도 대비 사용률 실시간 모니터링
- 80% 초과 시 사용자 경고 토스트
- QuotaExceededError 발생 시 오래된 회의 자동 정리 후 재시도
- 50건 회의 제한으로 예방적 용량 관리

#### 3.5.2 신규성 분석 (★★★☆)

- 클라이언트 사이드 전용 아키텍처 자체는 PWA/SPA 패턴
- 그러나 **AI 회의 분석 + 전체 데이터의 클라이언트 전용 보관**이라는 프라이버시 중심 설계는 경쟁사 대비 명확한 차별점 (Otter.ai, Fireflies 등은 모두 서버 사이드 처리)
- localStorage QuotaExceeded 자동 복구 + 예방적 경고의 결합은 기술적 특이점

#### 3.5.3 진보성 분석 (★★★☆)

- 아키텍처 설계 선택이므로 단독 특허로는 진보성이 약함
- 발명 1, 3과 결합하여 "클라이언트 사이드에서 수행되는..." 한정사로 활용하면 청구항 강화에 기여

## 4. 독립 청구항 초안 및 종속항 방향

### 청구항 1: Smart Token Strategy (핵심 청구항)

**[독립 청구항 1]**

컴퓨터로 구현되는 실시간 회의 분석 방법으로서,

(a) 음성 인식 수단에 의해 변환된 회의 발화 텍스트를 시간순으로 저장하는 단계;

(b) 기 설정된 분석 주기에 따라 자동으로 분석을 트리거하는 단계;

(c) 상기 분석 트리거 시, 토큰 전략 설정에 기초하여 분석 입력 텍스트를 구성하되, 스마트 전략이 선택된 경우에는:
  - (c-1) 직전 분석 결과에서 구조화된 요약 정보를 추출하여 제1 텍스트 블록으로 구성하고,
  - (c-2) 현재 시점으로부터 설정된 시간 범위 내의 최근 발화 원문을 제2 텍스트 블록으로 구성하고,
  - (c-3) 상기 제1 텍스트 블록과 제2 텍스트 블록을 결합하여 단일 분석 입력으로 합성하는 단계;

(d) 상기 합성된 분석 입력을 대규모 언어 모델(LLM) API에 전송하여, 요약, 현재 맥락, 미해결 질문, 실행 항목, 및 제안을 포함하는 구조화된 JSON 형식의 분석 결과를 수신하는 단계;

(e) 상기 분석 결과를 분석 이력에 추가하고, 다음 분석 주기에서 상기 (c-1) 단계의 입력으로 활용하여 연속적 누적 분석 체인을 형성하는 단계;

를 포함하는 실시간 회의 분석 방법.

**[종속항 방향]**

- 종속항 1-1: 상기 (c-3) 단계에서, 최근 발화가 존재하지 않는 경우 전체 발화 중 마지막 N개를 폴백으로 사용하는 방법
- 종속항 1-2: 상기 토큰 전략이 사용자 설정에 의해 전체(full), 최근(recent), 스마트(smart) 중 선택 가능한 방법
- 종속항 1-3: 상기 분석 결과에 기초하여 자동으로 회의 태그를 생성하는 단계를 더 포함하는 방법
- 종속항 1-4: 상기 모든 처리가 클라이언트 사이드 웹 브라우저에서 수행되며, 서버에 회의 데이터가 저장되지 않는 방법
- 종속항 1-5: 상기 분석 주기에 연동된 시각적 진행 게이지를 UI에 표시하는 방법

### 청구항 2: Function Calling 기반 AI 회의 상호작용

**[독립 청구항 2]**

컴퓨터로 구현되는 회의 중 AI 양방향 상호작용 시스템으로서,

(a) 실시간 회의 발화 텍스트, 현재 AI 분석 결과, 및 사용자가 축적한 인사이트를 포함하는 회의 맥락 정보를 동적으로 구성하여 AI 채팅 시스템 프롬프트에 주입하는 회의 맥락 주입부;

(b) 대규모 언어 모델의 Function Calling 인터페이스를 통해, 상기 AI가 회의 맥락에 기반하여 사전 정의된 복수의 함수 중 하나를 자율적으로 선택하여 호출할 수 있도록 하되, 상기 복수의 함수는:
  - 사용자 인사이트를 분석 맥락에 추가하는 제1 함수,
  - 메모를 회의록에 추가하는 제2 함수, 및
  - AI 분석을 재실행하는 제3 함수를 포함하는 함수 호출부;

(c) 상기 함수 호출 결과를 이벤트 버스를 통해 회의 관리 시스템에 전파하고, UI를 실시간으로 업데이트하는 상태 변경부;

를 포함하는 회의 중 AI 양방향 상호작용 시스템.

**[종속항 방향]**

- 종속항 2-1: 상기 제3 함수(rerun_analysis) 호출 시, 축적된 사용자 인사이트가 분석 프롬프트에 반영되는 시스템
- 종속항 2-2: 상기 채팅 시스템 프롬프트가 최근 N개의 대화 히스토리를 포함하여 멀티턴 대화를 지원하는 시스템
- 종속항 2-3: 파일 첨부 기능을 통해 외부 문서를 회의 맥락에 추가할 수 있는 시스템

### 청구항 3: 하이브리드 오타 교정 사전 학습 시스템 (핵심 청구항)

**[독립 청구항 3]**

컴퓨터로 구현되는 음성 인식 텍스트 교정 방법으로서,

(a) 음성 인식 엔진에 의해 생성된 발화 텍스트를 수신하는 단계;

(b) 클라이언트 측 저장소에 유지되는 오타 교정 사전을 참조하여, 상기 발화 텍스트에 포함된 기 등록된 오타 패턴을 실시간으로 교정하고, 교정된 부분을 시각적으로 구별 표시하는 단계;

(c) 사용자가 상기 발화 텍스트를 인라인 편집할 때, 편집 전후의 텍스트를 단어 단위로 비교하여 변경된 단어 쌍을 감지하고, 감지된 단어 쌍을 상기 오타 교정 사전에 자동 등록하는 제1 학습 단계;

(d) 기 설정된 주기마다 최근 N개의 발화 텍스트와 현재 오타 교정 사전을 대규모 언어 모델에 전송하여, 음성 인식 오류 패턴을 분석하고 추가 교정 규칙을 제안받아, 상기 오타 교정 사전에 병합하는 제2 학습 단계;

(e) 상기 제1 학습 단계 및 제2 학습 단계에 의해 누적 확장된 오타 교정 사전을 이후의 음성 인식 텍스트에 적용하여 교정 품질을 점진적으로 향상시키는 단계;

를 포함하는 음성 인식 텍스트 교정 방법.

**[종속항 방향]**

- 종속항 3-1: 상기 (b) 단계에서, 교정된 텍스트 부분에 원본 텍스트를 데이터 속성으로 저장하고, 사용자가 호버 시 원본을 확인할 수 있는 UI를 제공하는 방법
- 종속항 3-2: 상기 (d) 단계에서, AI에 기존 사전을 함께 전송하여 중복 교정 규칙 생성을 방지하는 방법
- 종속항 3-3: 상기 오타 교정 사전이 클라이언트 측 localStorage에 저장되어 서버 전송 없이 개인화되는 방법
- 종속항 3-4: 상기 (c) 단계에서, 단어 수가 동일한 경우에만 단어 쌍 비교를 수행하여 오탐(false positive)을 최소화하는 방법
- 종속항 3-5: 사용자가 오타 교정 사전을 조회, 개별 삭제, 또는 전체 초기화할 수 있는 관리 인터페이스를 포함하는 방법

- 종속항 4-2: 상기 참석자 수집 단계에서, 수동 입력된 이름을 자동으로 연락처에 등록하는 방법

### 청구항 5: 클라이언트 전용 회의 데이터 관리 시스템

### 6.3 핵심 차별화 요약

Meeting AI의 가장 강력한 차별화 포인트는:

1. **"서버 없는" AI 회의 분석**: 모든 경쟁사가 서버 사이드 처리를 필수로 하는 반면, Meeting AI는 브라우저에서 직접 LLM API를 호출하여 데이터 주권을 사용자에게 완전히 위임
2. **누적적 분석 체인**: 회의가 길어질수록 토큰 비용이 선형 증가하는 경쟁사 대비, 이전 요약 활용으로 비용을 상수로 유지하면서 맥락 품질은 유지
3. **자기 학습 오타 교정**: 사용하면 사용할수록 교정 품질이 향상되는 개인화된 교정 시스템은 경쟁사에서 확인되지 않음

## 7. 선행기술 회피 전략

### 7.1 회피 대상 선행기술

| # | 선행기술 | 회피 전략 |
|---|---------|----------|
| 1 | 일반적 회의 전사+요약 특허 (Otter, MS 등) | 청구항에 "이전 분석 결과의 구조화된 요약과 최근 원문 발화의 결합"을 **필수 구성요소**로 명시하여 단순 요약과 구분 |
| 2 | ASR 오류 교정 특허 (US20070208567A1 등) | "음성 인식 모델을 변경하지 않고 텍스트 레벨에서 사전 기반 교정"임을 명시. "사용자 편집 + AI 제안의 이중 학습"을 필수 구성요소로 한정 |
| 3 | LLM 기반 대화 시스템 특허 | "회의 진행 중 실시간으로 회의 관리 시스템의 상태를 변경"하는 Function Calling 적용을 한정 |
| 4 | 클라이언트 사이드 데이터 관리 특허 | "AI 회의 분석 데이터의 클라이언트 전용 관리"와 "자동 정리 + 용량 경고"의 결합으로 한정 |

### 7.2 청구항 작성 시 회피 유의사항

1. **상위 개념 기재 회피**: "AI로 회의를 분석하는 방법" 같은 상위 개념은 선행기술에 의해 거절될 가능성이 높으므로, 구체적인 데이터 흐름과 처리 단계를 명시
2. **결합 발명 강조**: 개별 구성요소보다 "A + B + C의 결합에 의한 시너지 효과"를 강조하여 진보성 주장을 강화
3. **기술적 효과 명시**: 명세서에 토큰 사용량 감소율, 오타 교정률 향상 등의 정량적 효과를 실험 데이터로 뒷받침 (출원 전 실험 필요)

### 8.3 출원 전략 옵션

#### 옵션 A: 2건 분리 출원 (권장)

- **출원 1**: 발명 1(Smart Token Strategy) + 발명 2(Function Calling) 결합
  - 독립 청구항 2건 + 종속항 8~10건
  - 핵심: "연속적 누적 분석 + AI 양방향 상호작용"

- **출원 2**: 발명 3(하이브리드 오타 교정) 단독
  - 독립 청구항 1건 + 종속항 5~6건
  - 핵심: "사용자+AI 이중 학습 기반 오타 교정"

#### 옵션 B: 단일 출원

- 5개 발명을 1건의 출원으로 통합
- 비용 절감 가능하나, 발명의 단일성 요건에 의해 분리 요구 가능성

#### 옵션 C: 3건 분리 출원 (공격적)

- 출원 1: 발명 1 (Smart Token Strategy)
- 출원 2: 발명 2 + 3 (AI 상호작용 + 오타 교정)
- 출원 3: 발명 4 + 5 (회의 준비 + 스토리지)

### 8.4 KIPO BM특허 심사기준 적용

#### 발명의 성립성 (특허법 제2조)

한국 특허법상 컴퓨터 관련 발명이 특허 대상으로 인정받으려면:

1. **하드웨어와의 결합**: 소프트웨어에 의한 정보처리가 하드웨어를 이용하여 구체적으로 실현되어야 함
   - ✅ Meeting AI: 브라우저(하드웨어 자원), Web Speech API(마이크 하드웨어), localStorage(저장 하드웨어)와 결합

2. **기술적 사상**: 자연법칙을 이용한 기술적 사상이어야 함
   - ✅ 토큰 최적화에 의한 API 비용 절감, 오타 교정에 의한 텍스트 정확도 향상 등 기술적 효과 명확

3. **구체적 수단**: 정보처리의 구체적 수단 또는 방법이 기재되어야 함
   - ✅ 소스 코드 수준의 구체적 알고리즘 기재 가능

#### 청구항 기재 형식

KIPO가 인정하는 5가지 기재 형식 중 적용:

| 형식 | 적용 가능 여부 |
|------|:------------:|
| 방법 청구항 | ✅ (주 형식) |
| 물건(장치/시스템) 청구항 | ✅ (보조 형식) |
| 프로그램 기록 매체 청구항 | ✅ (선택) |
| 매체에 저장된 컴퓨터프로그램 청구항 | ✅ (선택) |
| 데이터 기록 매체 청구항 | △ (해당 없음) |

**권장**: 각 독립 청구항을 **방법 청구항 + 시스템 청구항** 듀얼로 기재하여 보호 범위 극대화

## 9. 결론 및 권고사항

### 9.1 종합 평가

Meeting AI 프로젝트는 **BM특허 출원 가치가 높은** 것으로 판단됩니다.

- **가장 강한 발명**: 발명 3 (하이브리드 오타 교정) — 선행기술 대비 명확한 차별화, 높은 진보성
- **가장 전략적 발명**: 발명 1 (Smart Token Strategy) — 경쟁사 대비 핵심 차별화 포인트, 방어적 특허로서 가치
- **보완적 발명**: 발명 2, 4, 5 — 종속항 또는 별도 출원으로 보호 범위 확장

### 9.2 즉시 실행 권고

1. **출원 전 실험 데이터 확보**
   - Smart Token Strategy의 토큰 사용량 비교 실험 (Full vs Smart)
   - 오타 교정 사전 누적에 따른 교정률 변화 추이 측정
   - 이 데이터를 명세서에 포함하면 진보성 주장 강화

2. **변리사 선임 및 명세서 작성 착수**
   - SW/AI 전문 변리사 선임 권장
   - 본 보고서의 청구항 초안을 기반으로 상세 명세서 작성

3. **선행기술조사전문기관 의뢰**
   - KIPRIS를 통한 정밀 선행기술 조사 위탁
   - 우선심사 요건 충족을 위해 전문기관 보고서 확보

4. **PCT 국제출원 고려**
   - 한국 출원 후 12개월 이내에 PCT 출원을 통해 미국(USPTO), 일본(JPO) 등 주요국 출원 확보 가능
   - 투자 유치 시 지식재산 포트폴리오로 활용
